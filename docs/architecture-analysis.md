### 2.4 架构设计原则（来自README）

仓库遵循**4大核心原则**:

#### 原则1: 结构化优先

**Why**: 便于解析、渲染、埋点、复盘

**How**:
- PRD中的API设计 → 标准化接口定义
- 埋点方案 → 事件名/参数结构化
- 用户流程 → 步骤编号 + 决策树

**示例** (PRD中的埋点设计):
```markdown
event: conversation_start
params:
  - user_id: string
  - session_id: string
  - trigger_source: enum[homepage, history, notification]
```

#### 原则2: 可验证

**Why**: 每个结论需要证据支撑

**How**:
- 市场数据 → 引用来源
- 用户痛点 → 用户访谈证据
- 技术可行性 → PoC或参考案例

**示例** (BP中的市场规模):
```markdown
TAM: 1000亿+ (职场心理健康市场)
证据: 艾瑞咨询《2024年中国心理健康行业报告》
假设: 渗透率10% → SAM 100亿
验证计划: 用户访谈100人，验证付费意愿
```

#### 原则3: 隐私与安全

**Why**: 情绪/心理场景的敏感性

**How**:
- 最小化数据收集
- 明确数据边界
- 危机内容优先安全兜底

**示例** (PRD中的隐私策略):
- 对话记录仅保留结构化摘要，不存原文
- 高风险关键词触发人工客服
- 用户可随时删除所有数据

#### 原则4: 可落地

**Why**: 设计必须能转化为工程任务

**How**:
- API定义 → 可直接对接后端
- 数据结构 → 可直接建表
- 实验计划 → 可直接执行
- 里程碑 → 可直接排期

**示例** (PRD中的API设计):
```markdown
POST /api/conversation/start
Request Body:
  {
    "user_id": "uuid",
    "context": "string"
  }
Response:
  {
    "session_id": "uuid",
    "first_question": "string"
  }
```
