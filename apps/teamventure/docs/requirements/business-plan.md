## 2. 市场与机会（Market & Opportunity）

### 2.1 目标用户

**ICP-1（P0优先）**：中型企业HR/行政
- 企业规模：50-200人
- 行业：互联网/科技/咨询
- 角色：HR/行政/HRBP
- 年龄：25-35岁
- 团建频次：每年2-4次
- 痛点：缺少团建经验、预算有限（人均<1000元）、考核压力大

### 2.2 Job Story

**当**我作为HR需要为50人团队策划一次2天1夜的团建，
**但**我既缺少团建经验又不想支付高昂的第三方服务费，
**以便于**我能在15分钟内获得3套完整方案（预算/行程/供应商），
**这样**我可以从"焦头烂额的协调者"转变为"手握多套方案的决策者"，成本降低50%，效率提升3倍。

### 2.3 市场规模（TAM/SAM/SOM）

**TAM（总市场）**：¥500 亿+
- 定义：中国企业团建服务市场（估算）
- 计算：5000万家企业 × 20%有团建需求 × 年均5万元/企业
- 参考：团餐市场2.2万亿元（2023），团建是其子集

**SAM（可服务市场）**：¥150 亿
- 筛选条件：50-500人规模企业（最强团建需求）
- 计算：500万家目标企业 × 年均3万元团建支出
- 渗透率假设：初期1%

**SOM（可触达市场，12-24M）**：¥5000 万
- 计算逻辑：
  - 目标：500家付费企业
  - 客单价：¥5,980/年（专业版）
  - ARR：500 × ¥5,980 = ¥299万（第1年）
  - 增长到5000家：¥2,990万（第2-3年）
- 里程碑口径：12个月达到 **500家付费企业 / MRR ¥50万**

**市场增长率**：20-30% CAGR（估算，参考团餐市场22.94%）

### 2.4 市场趋势（Why Now）

1. **团建需求强劲增长**
   - 50%企业人均预算>600元
   - 周边团建需求同比增长124%（携程2021数据）
   - 疫情后企业更注重团队凝聚

2. **传统服务商模式痛点明显**
   - 成本高：人均1000-2000元（含20-30%服务商利润）
   - 效率低：方案沟通周期3-7天
   - 不透明：价格、供应商信息不透明

3. **AI技术成熟可用**
   - LLM生成质量达到商用水平（GPT-4, Claude）
   - AI Agent编排技术成熟（LangChain）
   - SaaS基础设施完善（Vercel, AWS）

4. **工具化市场空白**（最大机会）
   - 搜索"团建管理系统"主要返回团队协作工具
   - 现有工具（活动行/会小二）聚焦其他场景
   - **先发优势明显**，可快速占领市场心智

## 3. 问题深度拆解（Problem Deep Dive）

### 3.1 典型场景

**场景A：中型企业年度团建**
- **用户**：张敏，30岁，互联网公司HRBP，负责120人产品研发团队
- **需求**：Q2季度团建，预算4-5万，2天1夜，周边游
- **痛点**：
  - "去年外包给第三方，花了10万，员工觉得很无聊"
  - "自己组织过一次，协调场地、餐饮累死了，来回沟通1周"
  - "老板问我为什么这么贵，我也说不清楚成本构成"
- **现有尝试**：
  1. 百度搜索"北京团建" → 广告多，信息杂乱
  2. 找3家第三方服务商询价 → 报价差异大，沟通周期长
  3. 自己研究攻略 → 花3天时间，还是不确定
- **卡点**：**决策焦虑**（不知道哪个方案性价比最高）+ **协调成本高**

**场景B：创业公司快速团建**
- **用户**：李雷，28岁，创业公司COO，20人团队
- **需求**：项目完成庆功，预算2万，1天，周末紧急组织
- **痛点**：
  - 预算有限，不想外包（太贵）
  - 时间紧急，3天内要定下来
  - 没经验，不知道怎么策划
- **现有尝试**：
  1. 找携程企业游 → 产品标准化，不够个性化
  2. 微信群问朋友 → 推荐的地方太远或太贵
  3. 自己联系场地 → 不懂砍价，担心被坑
- **卡点**：**时间紧迫** + **信息不对称**

### 3.2 现有替代方案的不足

| 替代方案 | 优势 | 劣势 | 用户痛点 |
|---------|------|------|---------|
| **第三方团建公司** | 一站式服务 | • 成本高（人均1000-2000元）<br>• 沟通周期长（3-7天）<br>• 方案标准化 | "太贵了，还不够个性化" |
| **OTA平台（携程/美团）** | 资源丰富 | • 不提供策划服务<br>• 需自己组织<br>• 缺少智能推荐 | "只是订产品，还是要自己策划" |
| **活动行（报名管理）** | 报名流程完善 | • 不涉及团建资源<br>• 无方案生成<br>• 非团建垂直 | "只能管报名，不解决策划问题" |
| **会小二（场地预订）** | 场地资源丰富 | • 仅提供场地<br>• 无活动策划<br>• 无行程管理 | "订了场地，活动还得自己想" |
| **自己研究攻略** | 免费 | • 时间成本极高（3-7天）<br>• 信息不对称<br>• 决策焦虑 | "花3天研究，还是不确定" |

### 3.3 核心痛点总结

1. **成本高昂**：传统外包人均1000-2000元，中小企业负担重
2. **效率低下**：方案沟通周期3-7天，协调成本高
3. **信息不对称**：不知道合理价格，担心被坑
4. **决策焦虑**：选择太多或太少，不知道哪个性价比最高
5. **缺少经验**：HR不是专业策划师，不知道怎么组织

**核心机会**：用AI解决"策划难+成本高+效率低"的三大痛点

### 5.4 数据结构（核心表）

**表1：plans（方案表）**
```sql
CREATE TABLE plans (
  -- 说明：
  -- - 对外暴露/跨系统传递的主键建议使用可排序的全局唯一ID（如 ULID/UUID），避免依赖自增ID
  -- - 若使用 PostgreSQL：推荐 TEXT/UUID + JSONB + CHECK 约束（下方示例为 PostgreSQL 风格）
  plan_id TEXT PRIMARY KEY,  -- e.g. plan_01JHxxxx（ULID）或 UUID
  user_id TEXT NOT NULL,
  plan_name VARCHAR(200),
  plan_type TEXT CHECK (plan_type IN ('budget', 'standard', 'premium')),
  people_count INT,
  budget_total DECIMAL(10,2),
  start_date DATE,
  end_date DATE,
  itinerary JSONB,  -- 行程表
  budget_breakdown JSONB,  -- 预算明细
  suppliers JSONB,  -- 供应商信息（或做关联表）
  status TEXT CHECK (status IN ('draft', 'confirmed')) DEFAULT 'draft',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```
